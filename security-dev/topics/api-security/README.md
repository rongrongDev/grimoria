# API Security — The OWASP API Top 10 Lens, Mass Assignment, Rate Limiting, and Contract Discipline

**Date:** 2026-07-06 · **Tier:** core (full depth) · **Standards:** OWASP API Security Top 10 **2023** (mapped section-by-section below); OWASP Top 10 2021 where it overlaps; CWE-915 (mass assignment), CWE-213 (excessive data exposure), CWE-770 (unbounded resource allocation) · **Standalone:** yes · **Related:** [../authorization/](../authorization/README.md) (API1/API5 in depth), [../oauth-oidc-jwt/](../oauth-oidc-jwt/README.md) (API2), [../ssrf-xxe-deserialization/](../ssrf-xxe-deserialization/README.md) (API7), [../supply-chain/](../supply-chain/README.md) (API10 adjacency)

APIs concentrate a specific risk profile: **every capability is machine-enumerable, machine-callable, and stripped of the UI's accidental guardrails.** The web app only *shows* your own invoices; the API will answer whatever it's asked. This doc covers what's *distinctive* about API security; the vulnerability classes APIs share with everything else live in their own topic docs, and this doc's coverage map routes to them.

**Coverage map (OWASP API Top 10 2023):** API1 BOLA and API5 BFLA → [../authorization/](../authorization/README.md) (full depth) + §1 below for the API-shaped review angle · API2 Broken Authentication → [../authentication-and-sessions/](../authentication-and-sessions/README.md), [../oauth-oidc-jwt/](../oauth-oidc-jwt/README.md) · API3 BOPLA → §2 · API4 Unrestricted Resource Consumption → §3 · API6 Sensitive Business Flows → §4 · API7 SSRF → [../ssrf-xxe-deserialization/](../ssrf-xxe-deserialization/README.md) §1 · API8 Security Misconfiguration → §5 + [../cloud-and-infra/](../cloud-and-infra/README.md) · API9 Improper Inventory → §6 · API10 Unsafe Consumption of 3rd-party APIs → §7.

## 1. Object/function authorization, API-shaped (API1, API5)

Full treatment in [../authorization/](../authorization/README.md); what the API context *adds*:

- **Every parameter is an enumeration handle.** Sequential IDs turn one BOLA into a full-table dump at API speed; the [access-log signature](../authorization/README.md) §2 (one principal, many objects) is your detection, and per-principal rate limits (§3) are the speed bump that makes it visible before it completes.
- **GraphQL erases the route-based mental model.** Authorization must live *per-node/per-field in resolvers*, not per-endpoint — a perfectly guarded `user(id:)` query coexists with a `node(id:)` interface or a nested `orders { customer { email } }` traversal that skips the check. Review the schema as the route inventory ([authorization](../authorization/README.md) §3) and enforce policy in one resolver layer, not in each resolver's good intentions.
- **Batch endpoints multiply misses:** the bulk-get/bulk-update that loops over IDs *inside* the handler, after the per-request check already ran, is a classic single-check-many-objects gap. Each ID in a batch gets the same scoped-fetch/policy check as a single request.

## 2. Property-level authorization: mass assignment & excessive exposure (API3, CWE-915, CWE-213)

**Failure mode — write side (mass assignment).** Request body bound wholesale onto a model: `User.update(req.body)` — and the body arrives with `"role": "admin"`, `"email_verified": true`, `"price": 0.01`, or `"tenant_id": <someone else's>`. The framework did exactly what it was told; nobody told it which fields were *the user's to set*. This is [privilege escalation](../authorization/README.md) §4 delivered through a serializer, and it recurs because ORMs/frameworks make whole-body binding the path of least resistance.

**Failure mode — read side (excessive exposure).** Serializing the whole model out: internal flags, cost fields, other users' embedded objects, password hashes (it has happened, repeatedly, in public post-mortems) — shipped to every client "because the mobile app might need it," discoverable by anyone who opens the network tab. GraphQL variant: overly-connected schema exposing traversals nobody meant to offer (§1's nested-field problem, read edition).

**Detection.** Grep leads: body/params spread or passed directly into create/update/model constructors (`**req.body`, `update(params)`, `Object.assign(model, body)`, mass-assignable base classes); serializers that are the model (`to_json` on the record, missing serializer/DTO layer). Behavioral test: send a write with one extra privileged field; assert it's ignored/rejected. Diff every response schema against "what does this client *need*?"

**Fix.** **Explicit allowlists both directions**: input DTOs/schemas naming every writable field (strong parameters, serializer `fields=[...]`, zod/JSON-Schema with `additionalProperties: false`), output DTOs naming every readable field — the deny-by-default pattern ([mindset](../../principles/security-mindset.md) heuristic 3) applied to properties. Privileged fields (`role`, `verified`, `tenant_id`) change only through dedicated endpoints that check the *assigner's* privilege ([authorization](../authorization/README.md) §4).

**Prevention.** Lint/SAST rule for raw-body-to-model patterns (mechanically detectable in most frameworks); schema-first contracts (OpenAPI/GraphQL SDL) with `additionalProperties: false` and CI contract validation — unknown-field requests fail loudly in staging instead of silently in production; the one-extra-field test in the standard test scaffold for every writable resource.

## 3. Unrestricted resource consumption (API4, CWE-770)

**Failure mode.** No bounds on what one caller can ask for: page sizes of 10⁶, unbounded filters forcing full-table scans, file uploads without size caps, GraphQL queries of arbitrary depth/breadth (one crafted query = one outage — DoS by data-model traversal), N+1-triggering expansions, unmetered expensive operations (report generation, exports, LLM calls). Also the *economic* variant: your API fronting a paid downstream (SMS, maps, LLM tokens) with no per-caller budget — the attack is your invoice. And the *auth-adjacent* variant: unlimited attempts at login/reset/OTP endpoints, which is [authentication](../authentication-and-sessions/README.md) §2's stuffing surface — rate limiting is an authN control, not just an availability one.

**Detection.** For each endpoint, four questions: max request size? max response cardinality (page cap enforced *server-side*)? per-caller rate/concurrency limit? cost ceiling for the expensive paths (query depth/complexity limits for GraphQL, timeout + row limits for filters)? Load logs: p99 response sizes and slowest-query-by-endpoint tell you where the unbounded paths already are.

**Fix.** Pagination mandatory with server-enforced max page size; request-size and upload caps at the edge; per-principal (not just per-IP — [authentication](../authentication-and-sessions/README.md) §2's lesson) rate limits, tiered by endpoint cost; GraphQL depth + complexity budgets; timeouts and result caps on every query path; spend alerts and hard budgets on metered downstreams.

**Prevention.** Limits live in the API gateway/middleware layer as *default-on* config — a new endpoint inherits limits unless it opts out with justification ([mindset](../../principles/security-mindset.md) §5's default rule; the auditable-exception pattern again); load tests in CI for the known-expensive endpoints; 429 handling documented in the contract so clients implement backoff instead of retry storms.

## 4. Sensitive business flows (API6) — the abuse-by-design-use class

**Failure mode.** No vulnerability in any single request — the *flow* is the target, at machine speed: scalper bots draining limited inventory, signup farms harvesting promo credits, scraping-by-pagination of your entire catalog/user directory, OTP-triggering as an SMS-cost attack on you or a harassment vector against victims. Scanners are blind here by definition (every request is "legitimate"); [threat modeling](../../principles/threat-modeling.md) is where this class gets caught, by asking "what does this flow give an adversary *at scale*?"

**Fix/detection in practice:** identify the money/scarcity/identity flows explicitly (checkout, signup, referral, OTP, quote generation); instrument per-principal and per-fingerprint velocity on those flows specifically; add friction *proportional to abuse signal* (progressive challenges, verification steps for anomalous velocity — not CAPTCHAs on every user forever); make the abuse-relevant quantities (inventory holds, promo grants) transactional so races can't multiply them ([mindset](../../principles/security-mindset.md) §2's second-enforcement rule). **Prevention:** business-flow abuse gets a line in every feature threat model ("what if 10,000 of these per minute?"); velocity metrics + alerting on the sensitive flows shipped *with* the feature, not after the first incident.

## 5. Security misconfiguration, API edition (API8)

The API-specific instances of a class that [cloud-and-infra](../cloud-and-infra/README.md) owns broadly — each is one config line and one recurring breach: **CORS** reflecting arbitrary origins with credentials (`Access-Control-Allow-Origin: <echo of request Origin>` + `Allow-Credentials: true` = any website can act as its visitors against your API — allowlist origins exactly, never reflect); **verbose errors** shipping stack traces/queries/paths to clients (uniform error envelope in production; detail to logs, correlation-ID to client); **debug/management endpoints** in production builds (`/actuator`, `/debug/pprof`, GraphQL introspection+playground on prod — compile out or gate, per [authorization](../authorization/README.md) §3's kill-switch rule); **TLS laxity** (HTTP fallback, no HSTS on API hosts); **missing security headers** on API responses that render anywhere. **Prevention for the class:** configuration-as-code with a hardening baseline test in CI (assert prod config: introspection off, debug routes 404, CORS allowlist exact, error envelope uniform) — config drift is invisible until it's a finding; make it a red build instead.

## 6. Inventory: shadow, zombie, and version-skew APIs (API9)

**Failure mode.** The APIs you forgot: `/v1` still deployed with the pre-fix authZ ([authorization](../authorization/README.md) §3's version-skew), the "temporary" partner endpoint from 2023, staging exposed with prod data, the internal API that a mobile release quietly made load-bearing and public. **You cannot secure endpoints you don't know you serve** — inventory failure converts every other section's "fixed" back into "vulnerable, elsewhere." **Detection:** diff *served* routes (gateway config, router dumps, k8s ingresses, API-gateway logs — what actually answered requests last month) against the documented contract; the delta is the finding list. Gateway logs beat documentation every time. **Prevention:** all traffic through the gateway (unroutable = uninventoried = impossible by construction); contract-first with the served-vs-spec diff as a CI/periodic check; deprecation policy with *dates and deletion* — a deprecated-but-serving endpoint is a zombie with a nicer name; new-endpoint checklist includes the inventory/policy-manifest entry ([authorization](../authorization/README.md) §3's deny-by-default gate covers the authZ half of this automatically).

## 7. Consuming third-party APIs (API10)

**Failure mode.** Trust flowing the wrong way down an integration: webhook handlers that don't verify signatures (anyone who finds the URL forges "payment succeeded" — [mindset](../../principles/security-mindset.md) §2's table, funded row), third-party response data treated as clean (rendered unescaped, written to DB unvalidated, fed to interpreters — taint is about *origin*, and "it came from Stripe's API" is still not *your* data), redirect-following clients ([SSRF](../ssrf-xxe-deserialization/README.md) §1's redirect bypass, consumption edition), and partner-API outages cascading into your fail-open paths ([mindset](../../principles/security-mindset.md) heuristic 5). **Fix/prevention:** verify webhook signatures with the provider's scheme + reject stale timestamps (replay window), respond-then-process asynchronously; validate third-party responses against a schema at the boundary like any input; per-integration egress allowlists and credentials ([secrets-and-keys](../secrets-and-keys/README.md) §5); explicit fail-closed decisions per integration, written down.

## 8. Review drill (any API diff)

1. New/changed endpoint or resolver → policy-manifest entry, scoped fetch, batch items individually checked? (§1)
2. Writable fields explicitly allowlisted; response DTO explicit; one-extra-field test present? (§2)
3. Limits inherited (page cap, rate, size, depth)? Opt-outs justified? (§3)
4. Is this a money/scarcity/identity flow? Then the at-scale question, velocity instrumentation, and transactional invariants. (§4)
5. Contract updated, version inventory clean, old version's sunset date real? (§6)
6. Any new outbound integration → signature verification, response validation, fail-closed decision recorded? (§7)
