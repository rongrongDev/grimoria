# Secrets & Key Management — Sprawl, Rotation, and Least-Privilege Credentials

**Date:** 2026-07-06 · **Tier:** core (full depth) · **Standards:** CWE-798 (hardcoded credentials), CWE-522, CWE-321; OWASP Top 10 2021 A02/A05 adjacency; NIST SP 800-57 (key management lifecycles) · **Standalone:** yes · **Executable scan procedure:** [../../skills/secret-leak-scanner/SKILL.md](../../skills/secret-leak-scanner/SKILL.md) · **Related:** [../../principles/incident-response.md](../../principles/incident-response.md) §4 (leak response), [../cloud-and-infra/](../cloud-and-infra/README.md) (workload identity), [../cryptography/](../cryptography/README.md) (what the keys protect)

Credentials are the #1 real-world initial-access vector — not exploits ([mindset](../../principles/security-mindset.md) §3). That makes secret handling the highest-ROI topic in this KB: every dollar here buys more breach-prevention than a dollar anywhere else. The strategic insight that organizes everything below: **the goal is not hiding secrets better; it's having fewer, shorter-lived, narrower secrets — ideally none.** Every section is a step down that gradient.

## 1. Hardcoded secrets (CWE-798)

**Failure mode.** Credentials in source, config files, container images, or infra templates. The killer detail juniors miss: **git history is forever** — a secret committed once and "removed" in the next commit is still in every clone, every fork, every CI cache, and (for public repos) already harvested: scanners watch the GitHub firehose and use committed cloud keys within *minutes* — measured repeatedly, not folklore. Adjacent burial grounds with the same property: container image layers (`ENV AWS_SECRET...` in a Dockerfile persists in the layer even if a later layer unsets it), CI logs (`set -x` echoing env), crash dumps, `.env` files that were in `.gitignore` on the original repo but not the fork, notebook outputs, and Slack/ticket history ("here's the staging password").

**Detection.** Three-layer scanning, each catching what the others miss: **pre-commit** hooks (cheapest point — the secret never enters history); **CI** on every push (catches hook-less contributors); **scheduled full-history org scans** (catches everything that predates the program). Detection logic: high-entropy string classifiers + provider-specific signatures (`AKIA...`, `ghp_`, `sk-`, `-----BEGIN ... PRIVATE KEY-----`, `eyJ` JWTs) — signatures for precision, entropy for the unknown formats; interpretation guidance and the full signature/where-to-look tables live in [the secret-leak-scanner skill](../../skills/secret-leak-scanner/SKILL.md). Also detect *usage*: cloud providers' own leaked-key detection (AWS quarantine policies etc.) is a free backstop — but by the time it fires, you're in incident response.

**Fix — the part everyone gets wrong: a leaked secret is rotated, never deleted.** History rewriting (filter-repo) is cleanup, not remediation; assume copied the moment it was pushed. Response order ([incident-response](../../principles/incident-response.md) §4): rotate now → audit usage over the *full exposure window* (from first commit, not from discovery) → then clean history to stop re-detection noise. For the codebase: replace with runtime injection from a secret manager (§3); never with "encrypted in repo" as the end state unless the decryption key management is itself solved (SOPS-class tooling is legitimate; a passphrase in the wiki is not).

**Prevention.** Pre-commit + CI gate ([secure-sdlc](../../principles/secure-sdlc.md) §3 ranks this the single cheapest catastrophic-bug preventer); `.gitignore` templates org-wide including `.env*`; a paved-road secret manager so the *easy* path is the safe one ([mindset](../../principles/security-mindset.md) §5 — secure-by-default beats vigilance; developers hardcode when injection is friction); make test fixtures use obviously-fake values (`test-key-not-real-...`) so scanners can allowlist by pattern instead of by file (file-level allowlists rot into holes).

## 2. Secret sprawl and the inventory problem

**Failure mode.** Not one leaked secret — the *inability to answer* "what secrets exist, who holds them, what do they unlock, when did they last rotate?" Sprawl is the precondition that turns minor leaks into long breaches: during an incident, "rotate everything the attacker could have touched" ([incident-response](../../principles/incident-response.md) §3) is executable in an afternoon with an inventory and flatly impossible without one — the three-rounds-of-rotation war story in that doc was a sprawl failure, not a rotation failure. Sprawl's growth pattern: every integration adds a key; every "temporary" copy into a CI variable, a laptop, a second vault becomes permanent; every departed employee's knowledge of static secrets walks out un-revoked.

**Detection.** You detect sprawl by *trying to build the inventory*: enumerate secret manager contents, CI/CD variable stores (org + repo + environment scopes), cloud provider secret/parameter stores, k8s Secrets across clusters, `.env` presence on hosts, and long-lived cloud access keys (IAM credential reports age them for you). Every secret found outside the designated manager is a finding; every secret *in* the manager with no owner label or last-rotation date is half a finding.

**Fix/Prevention (they merge here — sprawl is fixed by structure, not cleanup sprints).** One designated secret manager per environment tier, everything else migrates or dies; every secret carries metadata: owner, purpose, blast-radius note, rotation date/policy; secrets injected at runtime (env-at-start or mounted, per platform norm) so app code and images stay secret-free; **access to secrets is itself logged and audited** — the vault's access log is your incident-scoping tool and your anomaly-detection feed (a CI job reading a secret it never read before is a signal). Quarterly: inventory diff + orphan hunt (secrets nothing reads anymore — delete; every dormant credential is free attacker inventory, same logic as dormant admin accounts in [authorization](../authorization/README.md) §4).

## 3. Rotation — designed in, or never done

**Failure mode.** Rotation that requires coordinated downtime, code deploys, or heroics doesn't happen; the org accretes multi-year-old credentials that everyone is afraid to touch ("we don't know what breaks if we rotate it" — which is also the confession that they don't know what it *unlocks*). Then a departure, a laptop theft, or a leak forces rotation under incident pressure, against systems never designed for it, at 2am.

**Fix — the two-key pattern makes rotation boring:** every consumer must tolerate **two valid credentials simultaneously** (old + new) so rotation is: mint new → roll consumers at leisure → revoke old → verify nothing broke (the vault access log again). If any system can't do this, that's the work item. **Prevention:** rotation policy tiered by blast radius — high-privilege/static keys quarterly-or-better, everything on a schedule *someone tests* (an untested rotation runbook is [threat-modeling](../../principles/threat-modeling.md) §4's unmitigated-in-costume, ops edition: rotate one production secret per month as a drill); alerts on credential age exceeding policy; and the real fix — §5's short-lived credentials, which delete the rotation problem instead of scheduling it.

## 4. Key management for cryptographic keys

Same lifecycle discipline as secrets, plus crypto-specific rules ([cryptography](../cryptography/README.md) owns *algorithm* choices; this section owns *custody*): keys live in KMS/HSM and **never leave** — services request operations (sign this, decrypt this), not key material; per-purpose keys (the signing key is not the encryption key is not the other service's key — one compromise, one blast radius; also enables per-purpose rotation and audit); key hierarchy (data keys wrapped by master keys — envelope encryption — so rotating the master doesn't mean re-encrypting the world); versioned keys with decrypt-old/encrypt-new support (the two-key pattern's crypto twin, and it's built into KMS products — use theirs, don't build it); separation of duties — the identity that *administers* keys is not the identity that *uses* them (a compromised app shouldn't be able to schedule its own key's deletion). **Detection:** grep for key material in code/config (`-----BEGIN` in anything that isn't a test fixture is a finding per §1); KMS/IAM audit for `kms:*` grants ([cloud-and-infra](../cloud-and-infra/README.md) wildcard pitfall); keys with no rotation schedule.

## 5. Least-privilege and short-lived service credentials

**Failure mode.** The service account with god-mode because scoping was friction during a deadline: the app that only reads one bucket holding `s3:*` on `*`; the CI job with org-admin because someone once needed it to create a repo; the database user who owns the schema running the app ([injection](../injection/README.md) §1's blast-radius layer, defeated). Compromise of any such identity — via SSRF's metadata theft, a leaked key, a supply-chain payload in CI — inherits the entire over-grant. Over-privilege is the *multiplier* on every other vulnerability class in this KB.

**Detection.** Per identity: diff *granted* vs *used* permissions — cloud IAM analyzers (access advisor/last-accessed data) automate exactly this diff; flag wildcards in any policy attached to a workload identity; flag static long-lived keys where the platform offers workload identity ([cloud-and-infra](../cloud-and-infra/README.md) owns the platform mechanics).

**Fix.** The gradient, in order of preference: **no secret at all** (platform workload identity: IAM roles, k8s service-account federation — credentials minted, scoped, and expired by the platform; nothing to leak, nothing to rotate) → **short-lived, narrow tokens** (hours, single-purpose) → **static secrets in a manager, two-key rotatable** (§3) → hardcoded (never). Migrate up the gradient opportunistically: every static key replaced by workload identity permanently deletes a §1–§3 liability.

**Prevention.** New-service checklist requires a permissions manifest ("this service reads bucket X, publishes topic Y — nothing else") reviewed like code; infra-as-code lint bans wildcard grants without a justification tag ([auditable exceptions](../injection/README.md), the KB's recurring pattern); quarterly granted-vs-used reconciliation with automatic flagging of the gap; CI pipelines get *per-pipeline* identities — the shared "CI god account" is the supply-chain attacker's favorite prize ([supply-chain](../supply-chain/README.md) §4).

## 6. Review drill (any diff touching credentials)

1. Any literal that could be a secret — including in tests, fixtures, and Dockerfiles? Run [the scanner skill](../../skills/secret-leak-scanner/SKILL.md); remember history and image layers.
2. New credential introduced: is it on the §5 gradient's highest rung the platform allows? Owner + rotation metadata recorded? Two-key rotatable?
3. New permission grant: matches the manifest? Any wildcard? Justification tag?
4. Anything logging near auth/token/key code: could a secret transit the log line? (Log redaction for known formats is a cheap durable control — add the signature list from §1 to the log pipeline's scrubber.)
5. Secret *read* paths: through the manager's audited API, or a side channel (env dumped to debug endpoint, config echoed at boot)?
