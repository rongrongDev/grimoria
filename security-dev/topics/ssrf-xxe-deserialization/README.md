# SSRF, XXE, and Unsafe Deserialization — The Confused-Deputy Classes

**Date:** 2026-07-06 · **Tier:** core (full depth) · **Standards:** OWASP Top 10 2021 A10 (SSRF), A08 (Software & Data Integrity Failures — deserialization); OWASP API Top 10 2023 API7; CWE-918 (SSRF), CWE-611 (XXE), CWE-502 (deserialization) · **Standalone:** yes · **Related:** [../../principles/security-mindset.md](../../principles/security-mindset.md) §2 (trust boundaries), [../cloud-and-infra/](../cloud-and-infra/README.md) (metadata endpoints, egress policy), [../injection/](../injection/README.md) (the sibling root cause)

These three share a root cause worth naming: **the server is made to act against itself as a confused deputy.** In SSRF your server *makes requests* on the attacker's behalf, with the server's network position and credentials. In XXE your XML parser *fetches and embeds* what the document tells it to. In unsafe deserialization your runtime *constructs objects and runs their behavior* as dictated by attacker bytes. The attacker never touches the target directly — your own components, operating as designed, do it for them. They also share prevention infrastructure (egress control, parser hardening, sandboxing), which is why they live in one doc.

## 1. Server-Side Request Forgery (CWE-918)

**Failure mode.** Any feature that fetches a URL influenced by users: webhooks, URL previews/unfurlers, importers ("import from URL"), PDF/image renderers that follow embedded resources, avatar-by-URL, monitoring "check my endpoint" features. The attacker supplies a URL resolving somewhere *your server* can reach but they can't: cloud metadata services (credential theft — the mechanism inside several famous breaches; see [../cloud-and-infra/](../cloud-and-infra/README.md) on IMDSv2), internal admin APIs, `localhost`-bound services, other tenants' pods. The request arrives with your server's IP, network position, and often its service mesh identity — "internal, therefore trusted" ([zero trust](../../GLOSSARY.md)'s counterexample made flesh).

Bypass shapes reviewers must know to *review the fix* (these are validation-evasion categories, not a how-to): alternate encodings of internal addresses (decimal/octal/IPv6-mapped forms of the same IP); hostnames that *resolve* to internal addresses — including re-resolution between validation-time and fetch-time (DNS rebinding: the check saw a public IP, the fetch got 169.254-anything); redirects — validated URL 302s to an internal one and the HTTP client follows; and scheme smuggling (`file://`, `gopher://` where the client library supports them). Any allowlist/blocklist fix that doesn't address resolution, redirects, and schemes is decoration.

**Detection.** Inventory the fetch surface: grep for HTTP-client constructions (`requests.get(`, `fetch(`, `http.Get(`, `axios(`) where any part of the URL traces to request data, stored user content, or webhook config; add the *indirect* fetchers — HTML-to-PDF renderers, image processors, link unfurlers, SSRF-by-file-format (SVG/XML with remote refs — overlaps §2). Runtime: egress logs are the ground truth — a web app connecting to the metadata IP or to RFC-1918 space it has no business with is a detection regardless of cause; alert on it.

**Fix.** Layered, because each layer has the bypasses above:
1. **Design out where possible:** fetch by ID from a server-side allowlist of destinations; don't accept URLs at all ([threat-modeling](../../principles/threat-modeling.md) §4 *eliminate* — the strongest disposition).
2. **Validate right when URLs are the feature:** scheme allowlist (`https` only); resolve the hostname yourself, check *every* resolved IP against a deny set (loopback, link-local including 169.254.169.254, RFC-1918, ULA/IPv6 equivalents), then **connect to the vetted IP** (pin resolution — closes rebinding); disable or re-validate on redirects; timeouts and response-size caps (SSRF doubles as an internal port-scanner and a DoS lever).
3. **Architect the blast radius down:** run fetchers in an egress-restricted segment/sidecar (the "URL-fetcher service" pattern — one hardened component does all outbound fetching, everything else is egress-denied); IMDSv2/metadata-service hardening on cloud hosts; internal services authenticate callers ([../authentication-and-sessions/](../authentication-and-sessions/README.md) §6) so a forged request still needs credentials.

**Prevention.** A vetted internal `safe_fetch` library is the durable control — lint rule: raw HTTP-client calls with non-constant URLs are banned outside it (same blessed-wrapper pattern as JWT verification and password hashing; one audit point). Egress-deny-by-default in infrastructure ([../cloud-and-infra/](../cloud-and-infra/README.md)) converts residual SSRF from breach to log line. Test suite: the validator rejects metadata IPs in the encodings above, rejects a redirect-to-internal, rejects `file://` — table-driven, in CI forever. Design review trigger: any feature spec containing "fetch," "import from URL," "webhook," or "preview" gets the SSRF conversation before implementation.

## 2. XML External Entities — XXE (CWE-611)

**Failure mode.** An XML parser with DTD/external-entity processing enabled parses attacker-influenced XML; entity expansion then reads local files into the document, makes network requests (SSRF via parser — §1's twin), or detonates expansion bombs ("billion laughs" — DoS via exponential entity nesting). The trap is *defaults*: several older parser stacks (Java's pre-hardening defaults being the historical worst offender) enabled external entities out of the box, so teams "not using entities" were vulnerable through parsers they didn't know they had. Your XML surface is bigger than your API: SVG uploads, DOCX/XLSX (zipped XML), SOAP, RSS ingestion, SAML (XML signatures — where XXE and signature-wrapping issues meet; use the platform SAML library, never hand-rolled), sitemap fetchers.

**Detection.** Inventory XML entry points including the disguised ones above. For each parser instantiation, check hardening explicitly: in review, the *absence* of parser-hardening config on a known-risky stack is the finding ([secure-code-review](../../principles/secure-code-review.md) §2 — read what's absent). SAST rules for unhardened `DocumentBuilderFactory`/`SAXParser`/`XMLReader`/`libxml` configs are mature and low-noise; enable them. Modern stacks (Python's stdlib since 3.7-era hardening, Go's encoding/xml which doesn't fetch external entities, .NET post-4.5.2 defaults) are safe-by-default — the audit question becomes "which of our components are *not* on the safe-by-default list?" (legacy Java, PHP with libxml options, C/C++ libxml2 direct use).

**Fix.** Disable DTD processing entirely where possible (`disallow-doctype-decl` equivalent — rejecting the DOCTYPE outright also kills expansion bombs); otherwise disable external general + parameter entities and set entity-expansion limits. Prefer JSON where you control the format choice — smaller grammar, no entity machinery ([mindset](../../principles/security-mindset.md) heuristic 6: complexity is attack surface). For file-format XML (SVG/OOXML): parse with the hardened config *and* treat the files as hostile generally (they're also XSS vectors when served inline — serve user SVGs with `Content-Disposition: attachment` or from a sandboxed origin).

**Prevention.** One hardened XML-parsing utility per codebase, lint-banned direct parser instantiation (the pattern once more); a CI test feeding each XML endpoint a DOCTYPE-bearing document and asserting rejection (that's a malformed-input test, not an exploit); dependency review flags new XML-parsing libraries for the hardening check.

## 3. Unsafe deserialization (CWE-502)

**Failure mode.** Attacker-controlled bytes fed to a deserializer whose format can encode *types and behavior*, not just data: Java native serialization (`ObjectInputStream` — gadget-chain RCE; the class behind multiple headline breaches), Python `pickle` (arbitrary code by design — `pickle.loads` on untrusted data is not "risky," it is an RCE endpoint), PHP `unserialize` (object injection), Ruby `Marshal`, .NET `BinaryFormatter` (deprecated by Microsoft for exactly this), YAML loaders in non-safe mode (`yaml.load` pre-safe-default, `YAML.load` in older Ruby — instantiates arbitrary objects from tags). The recurring delivery paths: cookies/hidden fields carrying serialized session or view state ("it's signed" — is the signature *verified before* deserialization, with a live key?), cache/queue payloads (attacker reaches Redis/queue through some other hole, plants a payload, your worker deserializes it — a chain amplifier: cache write access becomes RCE), and "internal" RPC formats that turn out to be reachable ([mindset](../../principles/security-mindset.md) §2's queue boundary).

**Detection.** Grep is genuinely effective here because the APIs are enumerable: `ObjectInputStream`/`readObject`, `pickle.loads`/`pickle.load`, `unserialize(`, `Marshal.load`, `BinaryFormatter`, `yaml.load(` without `SafeLoader`/`safe_load`, `eval(` on data (deserialization's honest cousin). For each hit, the two questions: can the bytes trace to anything attacker-influenceable (include cache/queue/DB per taint rules), and is there authentication *of the bytes* (HMAC verified before parse) rather than *of the channel*?

**Fix.** Preference order:
1. **Data-only formats + schema validation:** JSON/protobuf/CBOR parsed into plain structures, then validated against a schema, then *mapped* onto domain objects by your code. The deserializer never chooses types; the attacker's input can't encode behavior. This is the fix — everything below is mitigation for when you can't have it.
2. **Authenticate before parse:** HMAC/signature over the serialized blob, verified before the deserializer touches it, keys managed properly ([../secrets-and-keys/](../secrets-and-keys/README.md)). Converts "anyone can send bytes" to "only key-holders can" — still one leaked key from RCE, hence order.
3. **Constrain the deserializer:** type allowlists (Java serialization filters/`ObjectInputFilter`, restricted YAML loaders, `pickle` custom `find_class` allowlist if pickle is truly unavoidable between your own components) — deny-by-default on *types*.
4. **Sandbox the consumer:** deserializing workers run unprivileged, egress-denied, resource-limited, like any other hostile-parser farm ([../injection/](../injection/README.md) §3's sandbox rule).

**Prevention.** Lint/SAST ban on the API list above, allowlisted per-call-site with justification (auditable exceptions, again); architecture decision recorded: "wire formats are JSON/protobuf; native serialization does not cross process boundaries" — cheap to write at design time, near-impossible to retrofit; dependency alerting tuned for deserialization-gadget CVEs in your stack's libraries (they arrive as "update commons-X" and read as boring — the [dependency-cve-triager](../../agents/dependency-cve-triager.md) reachability question "do we deserialize untrusted data anywhere near this?" is exactly what scanners can't ask).

## 4. Shared review drill

For any diff adding a fetcher, parser, or decoder:

1. What does this component *do on behalf of* its input — fetch, resolve, expand, instantiate, execute? That capability list is the threat list.
2. Can the input trace to an attacker, including via storage, cache, queue, or file upload? (Taint origin, not channel.)
3. Is the component on this doc's safe-config list for its stack, and is the config asserted by a test rather than by memory?
4. What contains the failure — egress policy, sandbox, least-privilege identity? If the answer is "nothing," that's a finding even with perfect validation, because §1–§3's bypass histories say validation eventually loses one round ([mindset](../../principles/security-mindset.md) §4, layers that fail independently).
