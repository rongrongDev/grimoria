# k6 — load testing that measures the system, not the generator

**Applies to:** k6 1.x (post-1.0, Grafana k6) · **Last verified:** 2026-07-06 · JMeter note at the end.
**Standalone:** yes. Definitions: a *VU* is a simulated client; an *executor* defines how load is generated; *open model* = requests arrive at a set rate regardless of response times (real users); *closed model* = fixed VU pool, each waits for a response before continuing (self-throttling); *coordinated omission* = the flattering measurement error closed models produce under slowdown.
**Related principles:** profile design, open-vs-closed reasoning, what "passing" licenses — `quality-dev/principles/performance-and-load-testing.md` (read first; this doc is mechanics).

## Executor choice — the decision that decides whether your numbers mean anything

| Question you're asking | Executor | Model |
|---|---|---|
| "Can we survive N requests/sec at peak?" (capacity) | `constant-arrival-rate` / `ramping-arrival-rate` | **Open — default for capacity** |
| "Where is the knee/ceiling?" | `ramping-arrival-rate` climbing past target until thresholds break | Open |
| "How do N pooled workers/batch consumers behave?" | `constant-vus` / `ramping-vus` | Closed — only when reality is genuinely a fixed pool |
| "Does one user's journey work under ambient load?" | `per-vu-iterations` alongside an arrival-rate scenario | Mixed |

The ticketing-launch war story in `quality-dev/principles/performance-and-load-testing.md` — "500 VUs passed easily," four-hour launch-day brownout — is a closed-model artifact: when the system slowed, the VUs slowed with it, and the test measured the generator's politeness. Arrival-rate executors keep pushing like real traffic does.

```js
export const options = {
  scenarios: {
    peak_api: {
      executor: 'ramping-arrival-rate',
      startRate: 50, timeUnit: '1s',
      preAllocatedVUs: 200, maxVUs: 1000,   // VUs are a RESOURCE here, not the load definition
      stages: [
        { target: 400, duration: '5m' },    // ramp to 2× expected peak
        { target: 400, duration: '20m' },   // hold
        { target: 700, duration: '5m' },    // overload probe: find degradation behavior
        { target: 0,   duration: '5m' },    // recovery: does it come back?
      ],
    },
  },
  thresholds: {
    'http_req_duration{scenario:peak_api}': ['p(95)<300', 'p(99)<1200'],  // percentiles, never avg
    'http_req_failed{scenario:peak_api}':   ['rate<0.001'],
    'checks': ['rate>0.999'],
  },
};
```

If `maxVUs` saturates (k6 warns: "insufficient VUs"), the *generator* became the bottleneck and your results measure k6, not the target — raise the allocation or add load-generator instances before believing any number.

## Scripting the profile honestly

- **Endpoint mix from production telemetry** — weight scenarios per real traffic proportions (`quality-dev/principles/performance-and-load-testing.md`, "derive, don't invent"). Record the source query + date in the script header; an unversioned profile is folklore in six months.
- **Think time** (`sleep(1 + Math.random()*3)`) inside journey scripts — its absence turns each VU into a scripted DoS. (Yes, `sleep` is banned in correctness tests per `quality-dev/principles/flakiness.md`; in a load *profile* it is the model of the user. Different artifact, different rules — don't let the lint habit delete your think time.)
- **`check()` + the `checks` threshold**: a 200-with-empty-body storm is a real failure mode I've watched pass a status-only load test — check response *content* markers. Individual `check` failures don't abort (good: you want failure-rate data, not an aborted run); thresholds decide pass/fail at the end.
- **Parameterize data** (`SharedArray` of users/SKUs) so you're not measuring one cache line: 10k requests for the same product ID is a cache benchmark wearing a load-test costume.
- **Tag everything** (`tags: { name: 'GET /orders/:id' }`) so URL cardinality doesn't shred your metrics, and thresholds can target endpoint classes.

## Running it right

- **Never from your laptop** for real numbers: laptop NIC/CPU/VPN become the ceiling. Dedicated runners in the same cloud region-class as real clients (or k6 Cloud / distributed operator for >10k RPS class loads).
- **Environment statement in the output** (the verdict template from `quality-dev/principles/performance-and-load-testing.md`): target env size vs prod, data volume, cache state, neighbors. No statement ⇒ no sign-off.
- **CI placement** (`quality-dev/principles/ci-cd-integration.md`): a 2–3 min smoke profile (low arrival rate, tight thresholds) in the merge queue at most; full peak/overload/soak profiles scheduled nightly/pre-release. Load tests never gate PRs.
- **Soak** = same arrival-rate scenario, hours-long hold, thresholds on *trend* metrics (heap, pool wait, disk) via the target's own monitoring — k6 sees latency; leaks show up in the target's telemetry. A soak without target-side dashboards watches the symptom and misses the disease.

## Common pitfalls

| Pitfall | Consequence | Instead |
|---|---|---|
| Closed model for capacity questions | Coordinated omission; launch-day surprise | Arrival-rate executors (table above) |
| Thresholds on `avg` | Tail pain invisible (95% fast / 5% at 8s "passes") | `p(95)`, `p(99)` per endpoint class |
| No think time | Testing a DoS, not customers | Journey scripts with randomized think time |
| Single test datum | Measuring the cache | `SharedArray` parameterization |
| Status-only checks | Empty-200 storms pass | Content-marker checks + `checks` threshold |
| Laptop/VPN generation | Generator is the bottleneck | Dedicated same-region runners; watch `maxVUs` warnings |
| Stopping at target load | Cliff behavior unknown | Overload + recovery stages every run |
| 10-min test declared "soak" | Leaks/pool exhaustion ship | Hours-long holds, target-side trend thresholds |

## Failure mode → detection → fix → prevention

| Failure mode | Detection | Fix | Prevention |
|---|---|---|---|
| Flattering numbers vs prod pain | User complaints contradict green load tests | Re-run open-model, tails, prod-like env | Executor + threshold review checklist per new script |
| Generator saturation read as target ceiling | k6 VU warnings; generator CPU >70% | Scale/distribute generators | Generator-side metrics exported alongside target metrics |
| Profile drift from real traffic | Quarterly telemetry diff vs script weights | Re-derive mix | Profile header cites source query + refresh date; stale ⇒ CI warning |
| Dependency collateral damage | Third-party rate-limit bans during runs | Sandbox/coordinate dependencies | Run-book: dependency notification step before full profiles |

## If you're on JMeter

Same principles; JMeter's thread-group default is a **closed model** — use the Open Model Thread Group (5.5+) or arrival-rate plugins for capacity questions, or your results inherit coordinated omission. GUI is for building only; run headless (`-n -t`). Prefer k6 for new work: scriptable profiles code-review like code.
