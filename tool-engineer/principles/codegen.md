# Code Generation & Scaffolding — generated code is an interface with two consumers

**Applies to:** any code/config generator — API clients from OpenAPI/proto, ORM models from schemas, scaffolding templates, build-file generators. **Last verified:** 2026-07-06.

**The stance:** generated code has two consumers, and most generator authors design for neither. The first is the **build** — which will break org-wide if output semantics shift. The second is the **human reading a diff** — who must be able to tell "the generator changed" from "someone changed the inputs" from "someone hand-edited a file they shouldn't have." Every rule in this doc exists because one of those two consumers got hurt.

The incident that shaped my rules: our API-client generator had an undocumented dependency on the *iteration order of a JSON parse* to number positional fields. A patch upgrade of the JSON library changed that order. The generator ran green, the output compiled, and the fields were silently transposed — requests serialized with `user_id` in the `account_id` slot. Nothing failed until data corruption surfaced two days later. Every rule below (§1 determinism, §2 golden outputs, §3 version stamping) is a direct descendant of that week.

## 1. Determinism and idempotency: the first law

**Same inputs + same generator version → byte-identical output. Every time, on every machine.** Not "semantically equivalent" — byte-identical, because `git diff` is the only drift detector everyone already runs.

The usual sources of nondeterminism, all of which I have personally shipped and regretted:

- **Map/set iteration order.** Sort every collection before emitting. This is the #1 cause.
- **Timestamps in output.** `// Generated on 2026-07-06 14:32:01` guarantees every regeneration is a diff, which trains everyone to ignore codegen diffs, which is how the transposed-fields class of bug sails through review. Emit the *input hash* and *generator version*, never the wall clock.
- **Absolute paths** leaking from the build machine into headers or source maps.
- **Parallel generation with shared output** — two workers emitting into one file in completion order.
- **Locale/encoding drift** — sorting with the machine's locale instead of a fixed collation.

**Detection** is a one-liner every codegen setup must have in CI: regenerate, then `git diff --exit-code` (the drift gate). Run it twice back-to-back to separate *nondeterminism* (run 2 differs from run 1) from *drift* (both runs agree but differ from what's checked in — someone edited by hand or skew, §3–§4). The `codegen-drift-auditor` skill (`.claude/skills/codegen-drift-auditor/SKILL.md`) executes exactly this protocol and classifies the diff.

## 2. Silent breaking changes from the generator itself

When the *generator* upgrades, output can change with zero input changes — new formatting (noise), new field ordering (possibly semantic!), changed nullability or type mappings (breaking). The consuming team sees a wall-of-diff PR titled "bump codegen 3.1 → 3.2" and approves it on faith. That approval-on-faith is the failure.

Defenses, in order of value:

1. **Golden-output tests in the generator's own repo.** A fixture set of representative inputs and their exact expected outputs, committed. Any generator change that alters output must touch the goldens in the same PR — making output changes *visible and deliberate* instead of incidental. This is the single highest-leverage practice in codegen and the least practiced.
2. **A generator changelog that speaks about output**, not internals. "3.2: optional proto fields now map to `T | None` instead of `T`" is what consumers need; "refactored emitter visitor" is not.
3. **Semantic classification of golden diffs**: formatting-only / additive / breaking. Gate: breaking output changes require a major version of the generator (§3) and the breaking-change protocol from `tool-engineer/principles/distribution-and-versioning.md` §2.
4. **Regenerate-the-world in the generator's CI**: before releasing, regenerate the largest real consumer's code and compile/typecheck it. Expensive, and cheaper than the alternative.

## 3. Versioning generated code against the generator

Every generated file carries a header, emitted first, machine-parseable:

```
# Code generated by mytool-codegen v3.2.1 from schemas/user.proto (sha256:9f2c…). DO NOT EDIT.
```

Three fields, three jobs: **generator version** (detects skew — teammate on v3.1 regenerating v3.2 output produces churn diffs; CI pins the version, `tool-engineer/principles/distribution-and-versioning.md` §4), **input identity + hash** (detects stale output when inputs changed but nobody regenerated), **DO NOT EDIT** (the contract of §5, and the marker tools like Go's convention and linters key on).

**Check in generated code, or generate at build time?** The perennial fight. Decision tree:

- Consumers are humans reading/reviewing/debugging the code, or the generator needs a toolchain not everyone has, or you want drift visible in `git diff` → **check it in**, with the CI drift gate from §1 so it can't go stale.
- Output is huge, never read by humans, and the build system is hermetic with the generator version pinned as a build input (Bazel-class — `tool-engineer/extended/monorepo-build-tooling.md`) → **generate at build time**; checked-in copies would only rot.
- Default for internal tools: **check it in.** Reviewability and greppability beat repo size, and the drift gate makes staleness impossible rather than merely discouraged.

Never do the hybrid where some consumers check in and some generate fresh — you get two truths and the version-skew bugs of both.

## 4. Idempotent regeneration in practice: the noisy-diff death spiral

The adoption-killing failure mode: regenerating produces diffs unrelated to your input change. It starts small — a timestamp here, a reordered import there. Reviewers learn that codegen diffs are noise. Then a real semantic change hides in the noise and ships unreviewed. Then a team gets burned, declares the generator untrustworthy, and starts hand-maintaining "generated" files — and now you have §5's problem everywhere and no way back.

Break the spiral at step one: **zero-noise regeneration is a release-blocking requirement of the generator**, enforced by the double-regeneration check in §1. If a formatting change is genuinely wanted, ship it as its own generator release, regenerate the world in one dedicated PR ("3.3: formatting only, no semantic changes — verified by golden diff classification"), and keep it out of every other PR.

## 5. Manual edits to generated files

Someone will hand-edit a generated file. The only question is whether your design makes that safe, detectable, or catastrophic (their fix silently vanishes on the next regeneration — and it was a production hotfix).

Ownership models, in order of preference:

1. **Full generator ownership + extension points (default).** Generated files are never hand-edited; customization lives in separate hand-written files via the language's composition seams — partial classes, mixins/subclassing, sidecar modules the generated code imports if present, wrapper layers. Enforce, don't request: CI regenerates and diffs (any hand edit fails the gate), plus a lint that rejects PR diffs touching `DO NOT EDIT` files without a `[codegen]` commit tag.
2. **Protected regions** (`# BEGIN MANUAL / # END MANUAL` blocks the generator preserves). Legitimate for scaffolding that generates *starting points* (a new-service template generates once; the team owns the result — mark it "generated once, now yours" and drop the DO-NOT-EDIT header entirely). For continuously-regenerated output, protected regions are a maintenance trap: the generator must parse its own old output, merge semantics get hairy, and regions silently drop when markers are mangled. Allow only with round-trip tests (generate → hand-edit region → regenerate → assert edit survived).
3. **Mixed ownership without markers — never.** This is not a model, it's an incident schedule.

If you inherit model 3, migrate: checksum the checked-in files against fresh generation, extract the deltas into extension points, then turn on the CI gate. The `codegen-drift-auditor` skill's classification step finds the deltas for you.

## 6. Rolling out a generator change

A generator change is a distribution problem wearing a codegen hat — the full protocol lives in `tool-engineer/principles/distribution-and-versioning.md` and `tool-engineer/principles/adoption-and-rollout.md`. The codegen-specific rules:

- **One writer.** All regeneration for a given output tree goes through one pinned generator version and ideally one automated PR. Two teams (or two AI agents — `tool-engineer/orchestration/README.md` failure mode #3) regenerating with different versions produce oscillating diffs that erase each other's output.
- **Atomic regeneration.** When output format changes, regenerate *everything downstream in the same change*. Half-old half-new generated code is version skew inside a single repo.
- **Impact scan before breaking output changes**: consumers of generated code include hand-written code importing it. Dispatch `change-impact-scanner` to enumerate importers of the symbols that will change before you ship the generator release.

## 7. Failure modes → detection → fix → prevention

| Failure mode | Detection | Fix | Prevention |
|---|---|---|---|
| Silent semantic change in output (transposed-fields class) | Golden-output diff on generator upgrade; regenerate-the-world compile check | Pin generator; classify diff; fix or major-version the generator | Golden fixtures in generator CI (§2); generator changelog speaks output |
| Nondeterministic output | Double regeneration in clean env differs run-to-run | Sort collections; strip timestamps/paths; fix locale | CI runs generation twice + `git diff --exit-code` between runs |
| Noisy regeneration diffs → review blindness | Regeneration PRs with diffs unrelated to input change; reviewers rubber-stamping | Dedicated formatting-only releases; zero-noise gate | Drift gate blocks release of any generator producing incidental diffs (§4) |
| Manual edits lost on regeneration | Checksum checked-in vs fresh generation; "my fix disappeared" reports | Extract deltas to extension points; adopt ownership model 1 | `DO NOT EDIT` header + CI drift gate + PR lint on generated paths (§5) |
| Stale generated code (inputs changed, nobody regenerated) | Input hash in header ≠ hash of current inputs | Regenerate; find why the trigger didn't fire | Drift gate in CI on every PR touching inputs |
| Generator version skew between machines | Header version ≠ pinned version; oscillating diffs between teammates' PRs | Single pinned version in repo config; CI regenerates with the pin | Version pin file + CI as sole source of regeneration truth (§3, §6) |
| Scaffold template rot (new projects born broken) | Scaffold a project in CI weekly; build + test it | Fix template; regenerate-and-diff recently scaffolded projects | Scheduled scaffold-and-build CI job on the template repo |
