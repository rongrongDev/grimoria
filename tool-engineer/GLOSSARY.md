# GLOSSARY — tool-engineer

**Last verified:** 2026-07-06. One definition per term, for the whole KB. Docs link here; screens and help text should use these words the same way (`principles/internal-dashboards.md` §1). Where a term is contested in the industry, this entry states *this KB's* usage.

- **Actual adoption** — successful invocations per team per week on a current version. Contrast **assumed adoption**: installs, downloads, or the existence of a mandate. The gap between them is where tools die quietly. (`principles/adoption-and-rollout.md` §4)
- **Auto-update floor** — a server-side minimum client version, enforced with an actionable rejection at point of use. The only skew control that catches clients whose updater failed silently. (`principles/distribution-and-versioning.md` §3)
- **Autofix / codemod** — an automated rewrite that migrates code to comply with a rule or a new interface. An *autofix* rides the lint tool per-finding; a *codemod* is a one-shot repo-wide migration. Whoever ships the change writes the codemod. (`principles/static-analysis.md` §3, `principles/adoption-and-rollout.md` §3)
- **Baseline-and-ratchet** — rollout pattern for rules without autofix: snapshot existing violations into an ignored baseline, error on new violations immediately, fail CI if the baseline grows. (`principles/static-analysis.md` §3)
- **Blast radius** — the set of consumers (teams, repos, CI jobs, cron entries) affected by a change to a tool surface. Established by enumeration (the `change-impact-scanner` subagent), never by guess; the honest number is typically 2–5× the guess.
- **Breaking change** — any change requiring a consumer to act to stay working. For tools this surface includes command/flag names, **flag defaults**, exit codes, stdout/JSON shapes, config schema, env vars, and (for generators) output shape — not just APIs. (`principles/distribution-and-versioning.md` §1)
- **Canary channel** — a release channel (vs `stable`) that your team and volunteers ride for days of real use before promotion; dogfooding in mechanized, permanent form. (`principles/distribution-and-versioning.md` §5)
- **Clean-fixes-it** — the report "it works after `bazel clean` / cache reset." Never a user problem: it is the signature of an undeclared-input / cache-invalidation bug. (`extended/monorepo-build-tooling.md`)
- **Codegen drift** — divergence between checked-in generated files and what the pinned generator freshly produces. Classified into four diseases: nondeterminism, generator skew, stale output, manual edits — each with a different fix. (`principles/codegen.md`; `codegen-drift-auditor` skill)
- **Dark matter (consumers)** — call sites no scan can see: laptops, personal scripts, un-indexed repos. Sized via telemetry-vs-enumeration gap; mitigated by warning-first deprecation rather than hard cuts. (`.claude/agents/change-impact-scanner.md`)
- **Dogfooding** — the tool sitting in its own team's critical path (their deploys, their CI) for weeks, where its failures cost the authors. Demo-repo usage is rehearsal, not dogfooding. (`principles/adoption-and-rollout.md` §2)
- **Drift gate** — the CI job that regenerates all generated code and fails on any diff (`git diff --exit-code`). Makes staleness and manual edits impossible rather than discouraged. (`principles/codegen.md` §1)
- **Effective config** — the final resolved configuration after precedence (flag > env > project file > user file > default), printable with the *source of each value*. The `config show` command that deletes a support-thread genre. (`principles/cli-ux.md` §5-adjacent; `reference/click-typer.md` §5)
- **Exit-code contract** — the documented, tested mapping of exit codes to outcomes (0 full success / 1 runtime failure / 2 usage error). Load-bearing: scripts and CI branch on it. (`principles/cli-ux.md` §2)
- **False-positive budget** — the trust economics of a rule: each false positive spends trust ~10× faster than a true positive earns it; rules below ~95% true-positive rate on real code accumulate suppressions faster than fixes. (`principles/static-analysis.md` §1)
- **Generator skew** — different machines producing different generated output because they run different generator versions; presents as oscillating diffs between teammates' PRs. Fixed by an in-repo version pin with CI as the source of truth. (`principles/codegen.md` §3)
- **Golden output** — committed expected-output fixtures for a generator, such that any output-changing generator PR must visibly touch them. The main defense against silent breaking changes in codegen. (`principles/codegen.md` §2)
- **Golden path** — the officially supported, lowest-friction way to do a task with org tooling (e.g. "clone and run via the trampoline"). This KB covers the *tools* that pave it; the infrastructure underneath belongs to a platform-engineering KB.
- **Hermeticity** — a build action's output being a pure function of its *declared* inputs; no undeclared files, system tools, clocks, or network. Precondition for cache correctness. (`extended/monorepo-build-tooling.md`)
- **Manual-edit hazard** — hand changes inside generated files; regeneration silently destroys them. The one drift class where the fix is *stop, extract to extension points* — never regenerate first. (`principles/codegen.md` §5)
- **One-writer rule** — all regeneration of a given output tree flows through one pinned generator version and one serialized writer (human, bot, or agent). Prevents oscillating/conflicting codegen, especially under parallel agents. (`principles/codegen.md` §6; `orchestration/README.md` failure mode #2)
- **Protected region** — generator-preserved manual block (`BEGIN/END MANUAL`) inside generated output. Legitimate for generate-once scaffolds; a maintenance trap for continuously regenerated files. (`principles/codegen.md` §5)
- **Silent failure** — the tool did less than asked and exited 0. Worse than a crash: discovered downstream, after decisions were made on it. Partial success is failure unless explicitly opted into. (`principles/cli-ux.md` §2)
- **Staleness (dashboard)** — data presented without visible age, past its freshness expectation. A stale-but-green dashboard is worse than a down one: down redirects people to the source of truth; stale sends them away confident and wrong. (`principles/internal-dashboards.md` §2)
- **Sunset / tombstone** — the managed retirement of a tool: freeze → warn-with-pointer → active migration → white-glove the last 10% → brick loudly (the removed tool exits non-zero naming its replacement — the *tombstone*) rather than vanishing into "command not found". (`principles/adoption-and-rollout.md` §5)
- **Suppression cascade** — the trust-death sequence: inline suppressions → file-level disables → config-level disable → nobody reads any of the tool's output. Suppression counts per rule are the early-warning telemetry. (`principles/static-analysis.md` §1)
- **Three-part error contract** — every user-facing error states *what failed* (user's vocabulary), *why* (innermost cause, survived wrapping), *what to do next* (copy-pasteable). (`principles/cli-ux.md` §1)
- **Time-to-first-success** — minutes from "never used the tool" to first successful real invocation, measured, not assumed. The 15-minute bar decides stage-2 adoption. (`principles/adoption-and-rollout.md` §2, §4)
- **Tool adoption curve** — the staged rollout population sequence: dogfood → design partners → default-for-new → migration-of-existing → sunset-the-old, each with an exit criterion. Skipping stages relocates failure later and larger. (`principles/adoption-and-rollout.md` §1)
- **Trampoline (wrapper shim)** — a tiny checked-in script (`./tools/mytool`) that reads the repo's version pin, fetches exactly that version, and execs it — Gradle-wrapper style. Pins by construction; makes clone-and-run the real golden path. (`principles/distribution-and-versioning.md` §6)
- **Version skew** — different participants (teammates, CI, clients vs services) simultaneously running different tool versions. The steady state to design for, not an anomaly: pins, handshakes, floors. (`principles/distribution-and-versioning.md` §4)
- **White-glove migration** — hands-on, per-team migration of the last ~10% in a sunset; load-bearing because the stragglers are the genuinely weird cases that either expose real gaps in the new tool or fork the dead one. (`principles/adoption-and-rollout.md` §5)
- **Yank** — repointing "latest" away from a bad release in one reversible command, without rebuilding. Distinct from rollback-by-re-release, which is too slow for org-blocking tools. (`principles/distribution-and-versioning.md` §5)
