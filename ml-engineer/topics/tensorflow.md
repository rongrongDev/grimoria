# TensorFlow as the Alternative Framework

**Version 1.0 — 2026-07-06. Extended tier: production patterns + common pitfalls. Verified against TF 2.16–2.17 / Keras 3.x era; TF-specific serving via TF Serving / SavedModel.** Standalone. Related: [pytorch-training.md](pytorch-training.md) (the discipline transfers 1:1; this doc covers what *differs*), [../principles/train-serve-skew.md](../principles/train-serve-skew.md).

---

Everything in [../principles/](../principles/) is framework-agnostic — leakage, skew, eval, reproducibility discipline apply verbatim. This doc covers only where TF's shape changes the failure modes. Context for choosing: in 2026 PyTorch dominates research and most new production work; you'll meet TF primarily in **inherited production systems** (where its serving stack is genuinely strong and rewrites are rarely justified by framework preference alone) and **on-device/TFLite** pipelines. Analyze-existing-system work ([../guides/analyze-existing-ml-system.md](../guides/analyze-existing-ml-system.md)) hits TF constantly; migrations should be argued from maintenance cost, not fashion.

## Production patterns (TF's actual strengths — use them, don't fight them)

- **SavedModel as the skew killer.** TF's best production idea: preprocessing *inside* the exported graph. Keras preprocessing layers (`Normalization`, `StringLookup`, `TextVectorization`) are **adapted on training data** (`layer.adapt(train_ds)` — note: train split only; `adapt` on the full dataset is preprocessing leakage, [../principles/data-leakage.md](../principles/data-leakage.md) §3) and exported *in* the SavedModel. The serving payload is raw fields; normalization/vocab travel with the artifact. When you inherit a TF system, whether preprocessing is in-graph or Python-side is the first skew question to answer.
- **TF Serving:** mature, versioned model loading (directory-number convention), warm-swap on new versions, batching config. Pattern: version directories are immutable, written by CD only ([../principles/mlops-and-versioning.md](../principles/mlops-and-versioning.md)); `--enable_batching` with a tuned `batching_parameters` file for GPU throughput (then re-measure p99, [serving-patterns.md](serving-patterns.md) §2).
- **`tf.data` input pipelines:** the throughput lever. Canonical order: `shuffle(buffer) → map(parse, num_parallel_calls=AUTOTUNE) → batch → prefetch(AUTOTUNE)`. Profile with the TF Profiler's input-pipeline analyzer before touching the model — data starvation dominates "TF is slow" reports, same as PyTorch.
- **TFX/TFDV where inherited:** TFDV schema inference + drift detection maps directly to the data-validation gates of [../principles/testing-ml-systems.md](../principles/testing-ml-systems.md) §2 — if the system already has TFX, extend its schema/anomaly components rather than bolting on pandera beside them.

## Common pitfalls

- **`shuffle(buffer_size=small)` on ordered data = weak shuffling that looks fine.** A buffer of 1k on a 10M-row time-ordered file barely shuffles; training sees near-chronological order and metrics wobble mysteriously across "identical" runs. Shard-shuffle files, then row-shuffle with an honest buffer. (The `tf.data` twin of the sampler bugs in [distributed-training.md](distributed-training.md).)
- **`adapt()`/vocab computed on the wrong split** — the TF-specific spelling of preprocessing leakage; also its skew twin: recomputing vocab at *retrain* time changes token↔index mapping under any consumer that cached indices (embedding lookups downstream, logged features). Persist vocabs as versioned artifacts.
- **Training-mode flags:** `model(x, training=True/False)` and BN/dropout behavior — the eval-mode leakage from [pytorch-training.md](pytorch-training.md) §3 with different spelling; `.predict()` vs direct call differences around batching and dtypes have produced real parity failures. The golden-prediction test ([../principles/testing-ml-systems.md](../principles/testing-ml-systems.md) §4) against the *SavedModel via TF Serving*, not the in-process Keras model, is the only parity check that counts.
- **Retracing storms:** `@tf.function` retraces per new Python-value signature (e.g., Python ints as shape args, per-request unique shapes) — serving latency spikes that profile as compilation. Fix: `input_signature`, pad/bucket shapes.
- **Graph-mode silent numerics:** out-of-vocab silently mapping to OOV index 0/1 (monitor OOV rate — same UNK discipline as [../principles/train-serve-skew.md](../principles/train-serve-skew.md) §3), lossy dtype coercion at the SavedModel boundary (int64 Python → int32 graph), and `NaN`s propagating without eager-mode's friendly stack traces (`tf.debugging.enable_check_numerics()` in staging).
- **Keras 3 multi-backend drift:** Keras 3 runs on TF/JAX/PyTorch backends; "the same Keras code" is *not* the same numerics across backends — treat a backend switch as a model change with full re-eval, and pin the backend in the environment lock.
- **Checkpoint format confusion:** Keras `.keras`/H5 weights vs TF checkpoints vs SavedModel — teams archive one and need another; the registry entry ([../principles/mlops-and-versioning.md](../principles/mlops-and-versioning.md) §1) should store the *SavedModel* (deployable) plus training checkpoints (resumable), and the serialization round-trip test must load via the *serving* path.

**Migration judgment (TF→PyTorch, when asked):** justify by maintenance reality (team skill, dead TF-ecosystem deps, TFX operational cost) not preference; migrate *model by model* behind the golden-prediction parity test and a shadow phase ([../principles/deployment-and-serving.md](../principles/deployment-and-serving.md) §4); expect the feature/preprocessing port — not the model — to be where the risk lives (it's a train/serve-skew event by construction, [serving-patterns.md](serving-patterns.md) §5's migration note).
