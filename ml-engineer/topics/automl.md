# AutoML Tooling: Tradeoffs and Judgment

**Version 1.0 — 2026-07-06. Extended tier: production patterns + common pitfalls. Tool landscape (AutoGluon, FLAML, cloud AutoML services, Optuna-as-AutoML) as of this date.** Standalone. Related: [../principles/training-and-reproducibility.md](../principles/training-and-reproducibility.md) §3 (search discipline — AutoML is search at industrial scale), [../principles/evaluation.md](../principles/evaluation.md), [../principles/data-leakage.md](../principles/data-leakage.md).

---

AutoML automates the part of ML that was already the *most* automatable — model/hyperparameter selection — and none of the parts that actually sink projects: problem framing, leakage-safe features, split design, metric choice, serving, monitoring. Judged accordingly: it's a legitimate baseline accelerator and a dangerous substitute for evaluation judgment. My standing rule: **AutoGluon (or FLAML) on a correctly-split dataset is the first model I fit on any new tabular problem** — it sets the bar the bespoke work has to beat, in an afternoon. The failures come from letting it do more than that.

## Production patterns

- **AutoML as baseline oracle:** run it early, budget-boxed (`time_limit=1–4h`). If your hand-built model can't beat the AutoML ensemble, ship the boring one or question the feature set. If AutoML can't beat the *heuristic* baseline, the signal isn't in the data — a project-level finding worth a thousand tuning runs.
- **The split is YOUR job, always.** Hand AutoML pre-split data honoring the structure ([../principles/evaluation.md](../principles/evaluation.md) §2: time/group) — e.g. AutoGluon accepts explicit `tuning_data`; never let a tool random-CV time-series or grouped data because its default assumed i.i.d. Every AutoML tool's internal default is a random split; **this is the #1 way AutoML produces confident garbage.**
- **Constrain the deployment envelope upfront:** inference-latency/size constraints as search parameters where supported (AutoGluon `presets`/`infer_limit`), because the winning artifact is otherwise a 30-model stacked ensemble that can't meet your p99 ([../principles/deployment-and-serving.md](../principles/deployment-and-serving.md) §2) — distill it (`refit_full`, or train a single model on the ensemble's soft labels) as the standard exit path.
- **Same registry discipline as any model:** the AutoML run gets the closed record ([../principles/mlops-and-versioning.md](../principles/mlops-and-versioning.md) §1) — search budget, seed, tool version, data snapshot. "AutoML made it" is not lineage.
- **Cloud AutoML services (Vertex, SageMaker Autopilot):** buy speed-to-first-model and managed serving; pay in opacity (limited artifact introspection), egress lock-in, and per-prediction cost that compounds quietly. Fine for prototypes and low-QPS internal tools; for core high-QPS models, exportable-artifact tools (AutoGluon/FLAML) keep your serving and skew-testing options open.

## Common pitfalls

- **Leakage in = stellar metrics out, faster than ever.** AutoML maximizes whatever signal exists, including leaked signal — it will find the target-leaking feature you missed and build the whole ensemble on it. The too-good rule ([../principles/data-leakage.md](../principles/data-leakage.md) top) applies with a multiplier, and AutoGluon's feature-importance output is your first stop: one dominating feature → run the leakage protocol before celebrating.
- **Validation overfitting at tool scale:** thousands of internal fit/score cycles against the tuning split — the winner's reported validation score is biased upward *by construction* ([../principles/training-and-reproducibility.md](../principles/training-and-reproducibility.md) §3, winner's curse). Non-negotiable: a confirmation holdout the tool never touched, scored once. The gap you'll see (commonly 0.5–2% on tabular) is the tool's self-flattery, measured.
- **Default metric ≠ your metric:** tools default to accuracy/log-loss/RMSE; your objective is precision@K at an operating point with asymmetric costs ([../principles/evaluation.md](../principles/evaluation.md) §1). Set `eval_metric` explicitly or the search optimizes the wrong thing efficiently — and calibrate afterward regardless (stacked ensembles are frequently miscalibrated even when discriminative).
- **The unservable ensemble:** the leaderboard winner needs 2GB RAM and 400ms; nobody checked until deploy week. Envelope constraints upfront (above), and the deployment CD latency gate ([../principles/mlops-and-versioning.md](../principles/mlops-and-versioning.md) §2) as the backstop.
- **Retraining fragility:** re-running AutoML on next month's snapshot can select a *different architecture entirely* — your monitoring baselines, calibration, and explanation tooling all reset silently. For scheduled retraining, freeze the discovered pipeline (fixed architecture + hyperparameters from the original search) and retrain *weights only* on the cadence; re-open the full search deliberately, quarterly, through champion/challenger ([../principles/mlops-and-versioning.md](../principles/mlops-and-versioning.md) §4).
- **Skill atrophy framing:** the tool answers "which model"; it cannot answer "is this evaluation honest" — teams that treat the leaderboard as the deliverable ship the eval-protocol failures of [../principles/evaluation.md](../principles/evaluation.md) with extra confidence. The reviewer checklist for any AutoML-produced model is *identical* to a hand-built one; run the same skills (`data-leakage-scanner`, `eval-protocol-reviewer`).
