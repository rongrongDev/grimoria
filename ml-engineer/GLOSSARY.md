# Glossary — ml-engineer KB

**Version 1.0 — 2026-07-06.** Shared vocabulary for every doc, skill, and subagent in this KB. Terms link to their canonical doc; definitions here are deliberately short — the linked doc carries the judgment.

- **As-of join / point-in-time join** — Joining features to labeled examples such that each row sees only data available at its prediction timestamp. The mechanical defense against target leakage. → [principles/data-leakage.md](principles/data-leakage.md) §1, [topics/feature-stores.md](topics/feature-stores.md) §2
- **Availability time vs. event time** — When data *arrived and was usable* vs. when the event *happened*. Training joins on event time while serving sees availability time = staleness skew. → [principles/train-serve-skew.md](principles/train-serve-skew.md) §2
- **Batch scoring** — Precomputing predictions on a schedule into a table/cache; consumers look up. Minimal skew surface. → [topics/serving-patterns.md](topics/serving-patterns.md) §1
- **Champion/challenger** — Promotion gate: a new model (challenger) must beat the production model (champion) on the same frozen eval set, beyond the noise floor, without slice regressions. → [principles/mlops-and-versioning.md](principles/mlops-and-versioning.md) §2
- **Closed record** — The complete lineage a registered model must carry: code SHA, data snapshot ID, environment lock, config, split manifest, eval report. → [principles/mlops-and-versioning.md](principles/mlops-and-versioning.md) §1
- **Concept drift** — P(Y|X) changes: the same inputs now have different correct answers (adversaries adapt, behavior shifts). Detectable only through outcomes. Contrast *data drift*. → [principles/monitoring-and-drift.md](principles/monitoring-and-drift.md) §1
- **Confirmation set** — A second holdout scored exactly once by the winner of a search/selection, measuring how much the selection overfit the validation set. → [principles/training-and-reproducibility.md](principles/training-and-reproducibility.md) §3
- **Data drift (covariate shift)** — P(X) changes: inputs move while the learned relationship may still hold. Contrast *concept drift*. → [principles/monitoring-and-drift.md](principles/monitoring-and-drift.md) §1
- **Data leakage** — Information from outside the legitimate training scope (future, test set, or label) reaching training. Four classes: target leakage, train/test contamination, preprocessing leakage, group leakage. → [principles/data-leakage.md](principles/data-leakage.md)
- **Degradation ladder** — The written, product-approved sequence of fallbacks when a model can't serve: cached prediction → simpler model → heuristic → fail open/closed. → [principles/deployment-and-serving.md](principles/deployment-and-serving.md) §5
- **Effective batch size** — per-device batch × devices × gradient-accumulation steps; changing it changes the optimization problem (LR coupling). → [topics/distributed-training.md](topics/distributed-training.md)
- **Eval spec** — One page fixed before modeling: decision driven, operating point, one primary metric, guardrails, cost matrix, slices. → [principles/evaluation.md](principles/evaluation.md) §1
- **Feature store** — Infrastructure making two promises: one feature definition for training and serving; point-in-time-correct training sets. Offline store (history) + online store (low-latency) + materialization. → [topics/feature-stores.md](topics/feature-stores.md)
- **Feature staleness** — Age of a feature's value at serve time, vs. the freshness the training join assumed. → [principles/train-serve-skew.md](principles/train-serve-skew.md) §2
- **Feedback loop** — The model's decisions shaping its own future training data (only approved loans get repayment labels). → [principles/monitoring-and-drift.md](principles/monitoring-and-drift.md) §4
- **Golden-prediction (seam) test** — CI test scoring fixture entities through the training-side path and the serving path, asserting equal outputs. The skew integration test. → [principles/testing-ml-systems.md](principles/testing-ml-systems.md) §4
- **Group leakage** — The same entity (user, patient, account) on both sides of a split; the model memorizes entities and the metric counts it as skill. → [principles/data-leakage.md](principles/data-leakage.md) §4
- **Label maturity** — How long after prediction a label is considered final (chargebacks: weeks). Quality metrics must window by it or show phantom regressions. → [principles/monitoring-and-drift.md](principles/monitoring-and-drift.md) §2
- **Log-and-train** — Computing features only in the serving path, logging them, and training on the logs — skew becomes structurally impossible for logged features. → [principles/train-serve-skew.md](principles/train-serve-skew.md) §1
- **Loss-curve equivalence test** — 1-GPU vs. N-GPU run with identical effective batch and seeds must produce statistically indistinguishable curves; the unit test of distributed training. → [topics/distributed-training.md](topics/distributed-training.md)
- **Materialization** — The scheduled copy of feature values from offline to online store. Stalls silently; monitor freshness at serve time. → [topics/feature-stores.md](topics/feature-stores.md) §3
- **Model registry** — The gated catalog of deployable model versions, each a *closed record*; aliases (champion/challenger) moved only by pipelines. → [principles/mlops-and-versioning.md](principles/mlops-and-versioning.md) §1, [topics/experiment-tracking.md](topics/experiment-tracking.md) §2
- **Noise floor** — The metric variance of retraining the same model with different seeds; the minimum delta worth believing. → [principles/evaluation.md](principles/evaluation.md) §4
- **Off-policy evaluation** — Evaluating a new policy/model on data generated under the incumbent's decisions; structurally favors incumbent-like models. Why offline recsys gains vanish online. → [principles/evaluation.md](principles/evaluation.md) §3
- **Offline vs. online evaluation** — Metrics on held-out historical data vs. measured outcomes on live traffic (A/B). Offline gates what's worth testing; online decides what ships. → [principles/evaluation.md](principles/evaluation.md) §3
- **Online serving** — Predictions computed in the request path. Maximal freshness, maximal skew surface. → [topics/serving-patterns.md](topics/serving-patterns.md) §2
- **Point-in-time test** — CI test: an entity whose events all post-date the as-of timestamp must yield null/default features. Highest-value single test in an ML repo. → [principles/testing-ml-systems.md](principles/testing-ml-systems.md) §1
- **Preprocessing leakage** — Fitting transforms (scalers, encoders, selectors) on data that includes test rows before splitting. → [principles/data-leakage.md](principles/data-leakage.md) §3
- **PSI (Population Stability Index)** — Distribution-shift statistic per feature; investigate ≳0.1, alert ≳0.2. → [principles/monitoring-and-drift.md](principles/monitoring-and-drift.md) §2
- **Rolling-origin backtesting** — Forecasting evaluation: train to origin T, forecast T+1..T+h, roll forward, repeat; report across origins and horizons. → [topics/time-series-forecasting.md](topics/time-series-forecasting.md)
- **Shadow deployment** — New model scores live traffic; predictions logged, never acted on. Integration test against reality; blind to outcome quality. → [principles/deployment-and-serving.md](principles/deployment-and-serving.md) §4
- **Split manifest** — The split assignment persisted as a versioned artifact (IDs per split), not regenerated from a seed. → [principles/data-leakage.md](principles/data-leakage.md) §2
- **Target leakage** — A feature encoding future or label-derived information unavailable at prediction time. → [principles/data-leakage.md](principles/data-leakage.md) §1
- **Train/serve skew** — Any difference between features/preprocessing at training time vs. serving time. Three variants: divergent computation, staleness, schema/semantics drift. → [principles/train-serve-skew.md](principles/train-serve-skew.md)
- **Train/test contamination** — Test rows (or near-duplicates) influencing training: unstable splits, augmentation before splitting, repeated test-set consultation. → [principles/data-leakage.md](principles/data-leakage.md) §2
- **Winner's curse (selection bias)** — The best of N evaluated variants has an upward-biased reported score; corrected by confirmation sets. → [principles/evaluation.md](principles/evaluation.md) §4, [principles/training-and-reproducibility.md](principles/training-and-reproducibility.md) §3
