# Unity: Netcode — Stack Choice and Production Realities

**Applies to:** Unity 6.x; Netcode for GameObjects (NGO) 2.x, Netcode for Entities 1.x, Unity Transport 2.x; third-party (Fish-Networking, Mirror, Photon Fusion/Quantum) judged as of mid-2026 — this landscape moves fast, re-verify before committing.
**Last reviewed:** 2026-07-06.
**Principles this implements:** [../../principles/networking-and-multiplayer.md](../../principles/networking-and-multiplayer.md) — all model/prediction/desync reasoning lives there. This doc is *stack selection and Unity-specific traps*. Skill: `netcode-desync-reviewer`.

---

## 1. Stack decision tree (the question everyone asks first)

- **Co-op / casual, ≤16 players, GameObjects workflow, want engine-blessed:** **NGO**. Honest scope: server-authoritative *variable replication* with RPCs; it does **not** ship real client-side prediction/reconciliation for CharacterControllers out of the box — `NetworkTransform` interpolates, and "server-auth movement without prediction" means visible input latency. Plan to hand-roll prediction per [networking doc §3](../../principles/networking-and-multiplayer.md) on top, or accept host-migration-free listen-server feel for co-op.
- **Competitive action / high player counts / prediction as a core requirement, and ECS is acceptable:** **Netcode for Entities** — it has the real machinery (ghost snapshots + delta compression, prediction loop with rollback-and-resimulate, lag compensation, interest management) and it's the reason to accept DOTS's workflow costs ([dots doc §1](dots-jobs-and-burst.md) rung 3).
- **GameObjects workflow but NGO's ceiling is too low:** Fish-Networking (active, featureful, prediction support) or Mirror (battle-tested, huge install base) — third-party OSS carries maintenance-risk judgment: check commit recency and your team's fork-and-own appetite.
- **Deterministic lockstep/rollback (fighting, RTS) in Unity:** Photon Quantum (deterministic ECS sim engine, predict-rollback, its own fixed-point math + deterministic physics) — buying determinism as a product is *dramatically* cheaper than building it; the tax is their sim model + per-CCU pricing. Building it yourself in C#/Unity means everything in [networking doc §5](../../principles/networking-and-multiplayer.md) root-cause list is now your codebase's problem, including replacing PhysX ([unity README](README.md) physics note).
- **Relay/hosting:** UGS (Relay, Lobby, Multiplay) or self-hosted; listen-server + Relay is the cheap co-op path; dedicated Linux server builds for anything competitive ([security doc §2](../../principles/security-and-anti-cheat.md)).

## 2. NGO-specific traps (the ones that reach production)

- **`NetworkVariable` write permission defaults are server-only — good** — but `Owner`-writable NetworkVariables and owner-auth `NetworkTransform` are one checkbox away and quietly convert your game to client-authoritative. Every owner-auth surface is a [security doc §2](../../principles/security-and-anti-cheat.md) finding unless justified in writing. The `netcode-desync-reviewer` skill greps for exactly this.
- **RPCs are not a state model.** Teams build gameplay on RPC spam and rediscover why snapshots exist: late joiners miss every RPC-carried fact ("door is open" lives in an RPC that already fired). Rule: *state* lives in NetworkVariables/spawned state (replayed to joiners); RPCs carry *transient events* only (VFX triggers, one-shot notifications). Retrofitting this classification at month 12 is a rewrite; it's the first thing to check when auditing an NGO project.
- **Spawn/despawn ordering:** `OnNetworkSpawn` runs before other objects you reference may exist client-side; NetworkObject references arriving before their target spawns resolve null. Treat cross-object refs at spawn like the Awake/Start problem ([README lifecycle](README.md)) — defer to a post-spawn phase or use NetworkObject IDs + lookup-with-retry.
- **`NetworkBehaviour` + object pooling:** NGO owns spawn lifecycle; pooled NetworkObjects need `INetworkPrefabInstanceHandler`, and the [pooled-state-leak rule](../../principles/performance-and-frame-budgets.md) §4 applies to network state doubly — a pooled projectile carrying its previous owner's `OwnerClientId`-adjacent state is a cross-*player* bug, not just a visual one.
- **Bandwidth defaults:** NGO's per-variable delta + RPC overhead adds up; there's no interest management by default in the GameObjects path (network-visibility hooks exist but are DIY policy). At >8 players with naive replication you will blow the [networking doc §6](../../principles/networking-and-multiplayer.md) budgets — measure with the Network Profiler/RuntimeStats early, not at beta.
- **Scene management:** NGO's NetworkSceneManager owns networked scene transitions; mixing it with manual `SceneManager` calls desyncs client scene state (objects spawning into scenes the server doesn't think are loaded). Pick one owner.

## 3. Netcode for Entities — what to respect

The prediction loop re-simulates predicted ghosts every tick (rollback-and-replay per [networking doc §3](../../principles/networking-and-multiplayer.md), done for you) — which means **your prediction-group systems run up to N times per render frame**; a 2ms predicted-sim tick becomes 10ms+ under re-simulation at high RTT. Budget predicted-loop systems like rollback sim ([networking doc §1 rollback](../../principles/networking-and-multiplayer.md) — "sim must run ~10× real time"), keep them Burst-clean and allocation-free, and keep *presentation logic out of the prediction group* (it runs multiple times — the classic bug is a sound effect that plays 5 times, once per re-simulation; fire presentation from prediction only on `IsFirstTimeFullyPredictingTick`-style guards). Ghost relevancy/importance and delta compression give you the [§6 bandwidth](../../principles/networking-and-multiplayer.md) knobs; classify every ghost field as predicted/interpolated deliberately — interpolated-by-default fields on the local player's ghost reintroduce input latency piecemeal.

## 4. Determinism & desync posture in Unity (any stack)

Same-machine replay determinism is achievable with discipline (fixed tick via NGO/N4E network tick or your own accumulator; sim RNG streams; no `Update`-time gameplay). Cross-machine bit-determinism in managed C#/Burst floats is the hard road — [dots doc §3](dots-jobs-and-burst.md) — which is precisely why Quantum exists (fixed-point). Regardless of stack, implement the [networking doc §5](../../principles/networking-and-multiplayer.md) hierarchical per-tick checksum with your sim state; NGO gives you no desync detection because it doesn't know what your gameplay state is. The 4-player-only desync you eventually get will be found by the checksum bisect or not at all.

## 5. Test rig (Unity mechanics for [networking doc §7](../../principles/networking-and-multiplayer.md))

Unity Transport's simulator pipeline stage (or NGO's network simulator tooling) injects latency/jitter/loss in dev builds — wire the standard four-tier matrix into play-mode tests. Multiplayer Play Mode (Unity 6) runs multiple virtual players from one editor for iteration; **it is not a substitute for player-build multi-process tests on the CI box** — domain-shared editor state hides init-order and singleton bugs that real clients hit. Headless bot clients = the dedicated-server build + scripted `InputFrame` feed ([testing doc §2](../../principles/testing-and-determinism.md)); budget for this in sprint one of netcode work, it's the repro harness for everything after.
